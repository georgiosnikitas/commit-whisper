### Brain dump (George)
- **Spark / core insight:** Existing tools provide analysis and graphs from git history, but developers struggle to *understand* it. The wedge is **comprehension**, not data or visualization.
- **Users:** single developer, dev team, and IT/engineering manager of a dev team.
- **Aha moment:** the previously incomprehensible analysis/graphs become clear thanks to AI narrative explanation + guidance on how to work properly with git going forward.
- **AI + guidance:** value to dev, team, and IT manager — better commit messages, branching strategy, general git best practices.
- **Assets on hand:** none.
- **Stakes:** commercial product — "a product I am hoping people pay for." → High stakes; analysis calibrated to investor-grade rigor.

### Competitive grounding (background research, 2026-06-06)
- **Confirmed gap:** no tool today packages "narrate my repo's story in plain English + coach better git habits." OSS stats tools (git-quick-stats, gitstats, Hercules, onefetch) = stats only, no AI. Commercial eng-intelligence (LinearB, Swarmia, GitClear, Jellyfish, Pluralsight Flow) = team metrics dashboards, per-seat SaaS, manager-focused — not plain-language history or git coaching. AI git tools (aicommits, OpenCommit, Copilot) = forward commit messages only, not retrospective explanation.
- **Three tensions surfaced (need George's decisions):**
  1. **"Vitamin vs painkiller."** Risk that the product is educational/one-time and replicable with a Copilot/ChatGPT prompt over `git log` → thin moat. Need to identify the recurring, budgeted pain.
  2. **Who pays.** Money is overwhelmingly in team/manager analytics seats; solo devs rarely pay for personal git tooling. Tension with the "single developer" persona → likely freemium funnel for solo, paid for team/manager.
  3. **Surveillance risk.** Strong industry backlash against dev-productivity measurement (metrics theater). "Coaching" can read as surveillance → frame as dev-owned, not a manager dashboard.
- **Other risks:** commoditization by GitHub/Copilot ("explain history" could go native); privacy (sending commit data to an LLM blocks many orgs → offer local/Ollama/on-prem); accuracy (a hallucinated history narrative destroys trust).
- Full competitive digest preserved in `addendum.md`.

### Tension resolutions (George, 2026-06-06)
- **1. Vitamin → Painkiller.** Resolved via two arguments: (a) it runs from the terminal — the developer's home turf — and from **CI/CD**, so it can be run repeatedly or automated on a schedule (recurring, not one-time). (b) Teams/companies/even solo devs maintain **many repositories with very long histories that are impossible for a human to analyze** — scale-of-history-beyond-human-comprehension is the real painkiller.
- **2. Monetization.** Freemium: free tier limited by the number of most-recent commit messages analyzed. Paid tier (priced low — "a few euro") for those with long histories and many repositories, where value gained >> price paid.
  - ⚠️ OPEN (Mary push): pricing *unit* not yet defined (per repo? per seat? per month? per org?). "A few euro" may undervalue a team/company buyer. Revisit in Monetization/Scope.
- **3. Coaching vs surveillance.** Dev gets insights into their own work and how to improve. IT manager gets "transparency of the team's skillset and discipline on the given field."
  - ⚠️ OPEN (Mary push): the manager framing ("transparency of skillset and discipline") is exactly the wording that triggers the metrics-theater/surveillance backlash. Needs reframing toward team-health / risk / onboarding value rather than a per-developer scoreboard. Unresolved — flagged to George.
- **Working mode:** George is engaging substantively question-by-question → proceeding in **Coaching path** style (can switch to Fast draft on request).
- **Pending:** resolve the two OPEN pushes (pricing unit, manager framing), then proceed to draft sections.

### OPEN pushes resolved (George, 2026-06-06)
- **3a. Manager framing — RESOLVED toward team-health (safe framing).** The IT manager is *part of* the team (or many teams) and cares about **git repository health to help teams get better at their weaknesses and confirm their strengths.** Framing is improvement-oriented and team-level, NOT a per-developer surveillance scoreboard. This sidesteps the metrics-theater backlash. Lock this language into positioning.

Not blocking the brief, but must be addressed in Scope/Monetization as an open question (e.g., separate CI/server license or seat model for the team/org tier).

### Licensing RESOLVED — two-tier (George, 2026-06-06)

- This **resolves the CI/CD-vs-license contradiction**: the automation/team story lives in Tier 2; Tier 1 stays a clean solo desktop product. The 10x gap signals where the real value concentrates.

- ⚠️ OPEN (planted in brief, not blocking): **perpetual (one-time) vs subscription (per-year)** — biggest unresolved lever on revenue/valuation. To be decided in Monetization.
