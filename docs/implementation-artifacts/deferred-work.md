## Deferred from: code review of 1-4-local-repository-retrieval (2026-06-13)

- **Delimiter collision in the git-log parser** [src/retrieve/git-log.ts] — a commit message body (`%B`) that contains a literal RS (`\x1e`) or US (`\x1f`) byte would forge a record boundary or truncate fields (the parser splits naively on those bytes). Harden by realigning fields from the head and splitting RS only at a real boundary (RS immediately followed by a 40-hex sha), or switch the format to NUL (`%x00`) / `git log -z` framing. Tighten the `parts.length < 9` guard to `< 10` (the format always yields ≥10 fields) at the same time. Deferred: RS/US are non-printing controls that effectively never appear in real commit text, so this is a hardening pass, not a live bug.
- **No subprocess timeout** [src/retrieve/git.ts] — `execFileGitRunner` sets no `timeout`, so a git that blocks on a credential prompt, an fsmonitor daemon, or a wedged process hangs the CLI indefinitely with no recovery. Add an `execFile` `timeout` + `killSignal` and map a timeout to a `RetrieveError`. Deferred to pair with the Story 1.8 process shell and the Epic 5 remote-clone network-failure handling, where timeout/abort policy is load-bearing.
- **`spawn`-streaming for very large histories** [src/retrieve/git.ts, src/retrieve/git-log.ts] — the 256 MB `maxBuffer` both hard-caps huge monorepos (git is killed and all output discarded) and roughly doubles peak memory because `parseGitLog` does `stdout.split(RS)` on the whole materialized string. Stream the log line-by-line (or record-by-record) instead. A perf refinement already noted in the code comment; not needed for the walking skeleton.
- **Surface git's stderr / preserve failure specificity** [src/retrieve/git.ts, src/retrieve/local.ts] — the runner returns only stdout and the adapter flattens every read failure into one generic message, so a `maxBuffer` overflow, a permission denial, and a genuine repo error are indistinguishable to the user. The underlying error is already preserved as `cause`; richer stderr surfacing belongs with the `ui`/verbose-logging work (Epic 6) and the 1.8 shell.
- **Harden rename-path resolution** [src/retrieve/git-log.ts] — `resolveNumstatPath`'s greedy `^(.*)\{(.*) => (.*)\}(.*)$` regex plus the blanket `//`→`/` collapse can misfire on paths that legitimately contain braces, doubled slashes, or a literal ` => ` substring. Gate rename resolution on git's actual rename status (e.g. parse with `--raw`/status flags) rather than substring-matching the numstat path. Deferred: the current code handles the common brace/plain `old => new` forms (already documented as a simplification); exotic paths are rare and non-blocking.
